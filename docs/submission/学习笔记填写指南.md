## 你需要学什么

### 必须学懂

1. Mini-Drop 整体流程  
   `Web -> Server -> Agent -> Collector -> Analyzer -> Web`

2. Agent 是什么  
   Agent 是部署在 Linux 机器上的采集程序，负责心跳、接任务、执行采集、上报结果。

3. 状态机是什么  
   任务状态要从等待、运行、上传、完成或失败逐步变化，每次变化要记录原因。

4. perf 是什么  
   `perf` 是 Linux 性能采集工具，可以采样进程调用栈，后续生成火焰图。

5. 火焰图是什么  
   火焰图用来展示程序时间花在哪里，宽的函数表示占用时间更多。

6. eBPF 在这里干什么  
   eBPF 用来观察更底层的内核事件，例如 IO 延迟或调度抖动。

7. Analyzer 是什么  
   Analyzer 把原始数据转换成 Web 能展示的结果。

### 可以浅学

- React 页面细节。
- Go/Gin/GORM 具体写法。
- Docker Compose。
- MinIO 对象存储。
- LLM 智能归因。

### 暂时不用深学

- 完整 C++ Agent 实现。
- libbpf 深层机制。
- Java HPROF 深度分析。
- 多租户权限系统。
- 大规模分布式调度。

> 伞：我认为应该先从整体流程入手，把项目理解成“Web 下发任务、Agent 采集数据、Analyzer 生成图表、Web 展示结果”的闭环，再逐步学习每个模块的职责。

> 我的提问：Mini-Drop 这个题目我听不懂，到底要做什么？

### 学习心得 2：为什么需要 Agent，而不是普通脚本

#### 在做题过程中遇到的问题

> 我不理解为什么项目中要单独设计 Agent，而不是让 Server 直接执行 `perf` 或写一个普通脚本来采集数据。

#### 自己对问题的理解或预判

> 空：性能数据采集必须发生在目标 Linux 主机上，并且需要接触目标进程、系统权限、采集工具和本机环境。  
> 雨：如果 Server 直接采集，就要求所有目标进程都在 Server 所在机器上，不符合远程采集场景；如果只是普通脚本，也缺少心跳、任务状态、失败上报、超时控制和审计能力。  
> 伞：因此需要 Agent 部署在目标机器上，由 Agent 负责本地采集、心跳、任务执行和结果上报，Server 负责统一调度和管理。

#### 曾采取的行动或措施

> 我对照题目要求，重点阅读了 Agent 心跳、Server 离线判定、任务状态机、采集器执行等内容，并把 Agent 的职责整理为：心跳、接收任务、执行采集、上传结果、上报状态。

#### AI 对此问题的回答

> 使用工具：ChatGPT / Codex  
> 我的提问：Mini-Drop 为什么需要 Agent，而不是普通 workflow 或脚本？  
> AI 的回答：Agent 的价值在于它运行在目标机器上，能处理本地环境中的动态情况，例如 PID 是否存在、采集工具是否可用、权限是否足够、采集是否超时。普通脚本只能完成一次固定动作，而 Agent 可以持续心跳、接收任务、报告状态、处理失败，并和 Server 形成可观测的任务闭环。

#### 针对 AI 的回答，是否向导师进一步提问？

> 我认为 AI 的回答解决了我对 Agent 职责的基本疑问。但我还想进一步向导师确认：在 Mini-Drop 的简化实现中，Agent 是否必须做成长驻进程，还是可以先用定时轮询或命令行 worker 方式模拟 Agent，再逐步演进成长驻模式？
