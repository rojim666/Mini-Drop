# 00. Project Brief

## 一句话主旨

Mini-Drop 要做的是一个简化版 Linux 性能诊断平台：用户在 Web 上创建采样任务，Agent 在目标机器采集进程性能数据，Analyzer 生成火焰图和热点分析，最后在 Web 上展示结果。

## 用人话说它是什么

它不是普通网站，也不是单独写一个 `perf` 脚本。

它是一个小型平台：

- Web UI 负责让用户发起采集、查看 Agent、查看结果。
- Server 负责任务编排、状态落库、Agent 心跳、审计日志。
- Agent 负责在 Linux 机器上真正采集数据。
- Analyzer 负责把原始采集数据转换成火焰图、TopN 热点、建议。
- Storage 负责存放 `perf.data`、火焰图 SVG、JSON 结果。

## 生活类比

可以把它理解成一个“程序体检系统”：

- 被测进程是病人。
- Agent 是体检医生。
- `perf` / eBPF / py-spy 是体检仪器。
- Server 是前台调度。
- Analyzer 是报告生成器。
- Web UI 是体检报告页面。

## 考题真正想考什么

1. **端到端工程闭环**：能不能从用户点击开始，最终看到分析结果。
2. **Linux 性能采集能力**：能不能真实使用 `perf`、eBPF、用户态采集器。
3. **分布式系统基本功**：心跳、状态机、超时、错误处理、审计日志。
4. **数据分析与可视化**：能不能把原始数据变成可理解的图。
5. **交付能力**：README、设计文档、演示脚本、测试和提交历史。

## 我们的判断

题目允许合理取舍。正确策略不是一次性做满全部能力，而是：

1. 先跑通一条最小但完整的主链路。
2. 再补齐状态机、心跳、日志、测试这些工程骨架。
3. 然后选择 1-2 个扩展点做成亮点。
4. 最后视时间补智能归因或自然语言采集。

## 最小成功画面

演示时至少应该能看到：

1. Web 上出现一个在线 Agent。
2. 用户输入 PID、采样时长、采样率并创建任务。
3. 任务状态从 `PENDING` 变成 `RUNNING`、`UPLOADING`、`DONE`。
4. Agent 真正采集目标进程。
5. Analyzer 生成火焰图。
6. Web 任务详情页展示火焰图和 TopN 热点。
